# 2. Data formats expected by the pipeline

This document is the contract between *your data* and *the scripts*. If your
files match these formats, every script in the repository will work
unchanged.

---

## 2.1 UTM file — `raw_data/UTM_<ANG>_<REP>.txt`

Tab-separated ASCII export from the universal testing machine. The reader
(`data_processing.py::load_utm_data`) ignores the header block and starts
reading at the line that begins with `Time` (after the
`-----------Curve----------` separator).

```
Customer        SGCC_1.5
TestDate        4/4/2026
Coil No/Packet No       00-01
…                                   ← header, free-form
Lo(mm)  85                          ← gauge length (used by strain-rate check)
…
-----------Curve----------          ← required separator (verbatim)
Time    Load    Elong   Disp    Stress  Strain  TrueStress      TrueStrain
0       0.015   0.0000  0.002   0.477   0.000   0.477   0.000
0.999   0.304   0.0000  0.018   9.928   0.000   9.928   0.000
…
```

| Column   | Unit | Required by the pipeline |
| -------- | ---- | ------------------------ |
| `Time`   | s    | ✅ (strain-rate verification) |
| `Load`   | kgf  | ✅ (converted to N internally) |
| `Disp`   | mm   | ✅ (engineering strain via Lo) |
| others   | —    | ignored                  |

> The pipeline **does not** trust the `Stress` / `Strain` / `TrueStress` /
> `TrueStrain` columns produced by the UTM software. Stresses and strains
> are recomputed from `Load + Disp` using the geometry constants in
> `material_constants.json`.

**Specimen rename.** If your UTM exports are called `tensile_test_001.txt`,
just rename them so the angle and replicate are visible:

```
tensile_test_001.txt   →   UTM_00_01.txt
tensile_test_002.txt   →   UTM_00_02.txt
…
```

## 2.2 Ufreckles binary — `raw_data/<specimen>/<specimen>.res`

This is a MATLAB `-mat` file written by Ufreckles. The MATLAB scripts in
this repository read it with `load(file, '-mat')` and require the following
fields:

| Field      | Shape           | Meaning                                  |
| ---------- | --------------- | ---------------------------------------- |
| `xo`, `yo` | `[Nnod×1]`      | reference (un-deformed) node coordinates [px] |
| `conn`    | `[Nelt×4]`       | Q4 element connectivity                  |
| `U`       | `[2·Nnod × Nfrm]`| nodal displacements [px]; `[ux ; uy]`   |
| `pscale`  | scalar           | mm / px conversion (camera calibration)  |

If your DIC software exports a different format, you can mimic the
Ufreckles structure with a one-time MATLAB save:

```matlab
xo = ...; yo = ...; conn = ...; U = ...; pscale = ...;
save('00-01.res', 'xo', 'yo', 'conn', 'U', 'pscale', '-v7');
```

## 2.3 Ufreckles strain-gauge export — `raw_data/<spec>/<spec>-gage-NN.csv`

Per-zone strain history, exported by Ufreckles' "Strain gage" GUI. This
is the **secondary** input used by `extract_multizone_strains.m` and
ingested by `data_processing.py` for the analytical r-value and the
true-stress curve.

```
Result file;00-01.res
Reference image;img_0001.png
Strain gage;1
Small strain
Position X [pixel];Position Y [pixel]; Orientation; Length [pixel]; Width [pixel]
494.500000;1460.500000;90.000000;95.000000;55.000000
Filename;Step;Longitudinal strain [];Transverse strain [];Shear strain []
"img_0002.png";1;1.620e-05;6.712e-05;-1.036e-04
"img_0003.png";2;4.628e-05;1.302e-04;-7.985e-05
…
```

The reader auto-detects whether the delimiter is `;` or `,`.

| Column                    | Unit | Used as                  |
| ------------------------- | ---- | ------------------------ |
| `Step`                    | —    | image index (synced to UTM) |
| `Longitudinal strain []`  | —    | ε_yy (loading direction)  |
| `Transverse strain []`    | —    | ε_xx                      |
| `Shear strain []`         | —    | ε_xy (alignment QA)       |

Use **8 gauge zones** of size 95 × 55 px positioned as a 2×4 grid in the
straight gauge section. The repository ships a fully-populated example for
specimen `00-01` in [`examples/specimen_00-01/`](../examples/specimen_00-01).

## 2.4 Generated processed CSVs

After `data_processing.py` runs, every specimen yields two CSVs at the
repository root:

| File                       | Columns                              | Generated by             |
| -------------------------- | ------------------------------------ | ------------------------ |
| `<spec>.csv`               | `Step,Eyy,Exx,Exy,Eyy_filt,Exx_filt` | DIC mean strain history  |
| `stress-<spec>.csv`        | `Time,Load,Stress,Strain`            | UTM stress + true strain |

These are tracked in the repository so users can reproduce all downstream
plots without owning UTM/DIC hardware.

## 2.5 Material constants

`material_constants.json` is the **single source of truth** for elastic and
geometric properties:

```json
{
  "elastic":   { "E_MPa": 200000.0, "nu": 0.3, … },
  "anisotropy": { "r0": 0.712, "r45": 0.800, "r90": 0.742 },
  "yield":      { "sigma0_MPa": 312.35, … },
  "kinematic":  { "C1": 502.71, "gamma1": 499.72,
                  "C2": 100.37, "gamma2": 199.44 },
  "yld2000":    { "alpha": [ 1.05, 0.99, … ] }
}
```

Update **only this file** when porting to a different material — every
script reads it through `material_constants.py`.
